Web & API Automation Testing – LMS Dibimbing.id

Framework otomatisasi end-to-end untuk platform LMS Dibimbing.id, mengintegrasikan pengujian Web dan API (GraphQL) dengan pipeline CI/CD yang kuat serta kolaborasi tim secara real-time.

1. Problem

Platform LMS Dibimbing.id membutuhkan solusi pengujian otomatis yang menyeluruh untuk menjamin kestabilan alur pengguna krusial (Registrasi, Manajemen Karyawan, Manajemen Pelatihan) serta integritas data backend GraphQL. Pengujian manual dinilai tidak lagi efektif untuk siklus pengembangan yang cepat dan interaksi data yang kompleks.

2. Action

Saya merancang solusi otomatisasi terintegrasi menggunakan teknologi terbaru dan praktik terbaik industri.

Strategi yang saya terapkan:

  • End-to-End Web Automation: Membangun framework yang kokoh menggunakan Page Object Model (POM) dengan Selenium WebDriver untuk mengotomatisasi alur bisnis utama.

3. Result

  • Memperlancar koordinasi antara QA dan Developer melalui notifikasi Slack otomatis.
  • Memangkas waktu pengujian regresi secara signifikan tanpa mengurangi cakupan pengujian.
  • Menjamin integritas data backend yang tinggi melalui validasi GraphQL yang ketat.

Guru99 Bank Automation with Katalon Studio

Otomatisasi pengujian untuk aplikasi web Guru99 Bank menggunakan Katalon Studio. Mencakup fitur login, pendaftaran nasabah, pembukaan akun, transaksi, serta validasi data dengan struktur test case yang sistematis, pengujian berbasis data, dan pelaporan terintegrasi.

1. Problem

Pengujian manual pada situs web Guru99 Bank memakan waktu cukup lama dan rentan terhadap human error karena kompleksitas alur bisnis yang saling berkaitan—seperti login, pembuatan akun, transaksi, dan manajemen nasabah.

Setiap pembaruan sistem mengharuskan tim QA melakukan regresi manual berulang kali, sehingga siklus pengujian menjadi kurang efisien dan sulit dipantau secara konsisten.

2. Action

Membangun framework otomatisasi berbasis Katalon Studio untuk mengalihkan pengujian manual ke otomatis secara end-to-end.

Langkah-langkah yang saya lakukan:

  • Menyiapkan 55 test case otomatis dan mengelompokkannya ke dalam 5 test suite utama:
    • Smoke Test
    • Customer Management
    • Account Management
    • Negative Test Cases
    • Regression Test
  • Mengimplementasikan Custom Keywords untuk menjaga skrip tetap modular, dapat digunakan kembali, dan mudah dipelihara.
  • Menambahkan Test Listeners untuk secara otomatis mengambil screenshot pada kegagalan pengujian.
  • Menghasilkan laporan eksekusi secara otomatis dalam format HTML & PDF.
  • Mengoptimalkan delay, selector, dan struktur pengujian agar berjalan lebih cepat tanpa mengorbankan akurasi.

3. Result

  • Semua 55 test case dieksekusi dengan tingkat kelulusan 100%.
  • Waktu pengujian berkurang ~50% dibandingkan dengan metode manual.
  • Framework stabil dan siap untuk regresi dalam siklus pengembangan mendatang.
  • Dokumentasi pengujian lebih bersih dan lebih mudah diaudit berkat laporan otomatis dengan screenshot.

Sport Reservation API Automation Rest Assured

Solusi pengujian API otomatis untuk Sport Reservation API menggunakan Java, Rest Assured, dan TestNG. Mencakup modul autentikasi, kategori olahraga, dan CRUD aktivitas olahraga dengan pengujian yang deterministik, repeatable, serta pelaporan HTML yang informatif.

1. Problem

Tim QA membutuhkan laporan pengujian API yang jelas dan terstruktur untuk modul Auth, Kategori Olahraga, dan Aktivitas Olahraga.

  • Pengujian manual memakan waktu dan rentan terhadap kesalahan manusia.
  • Tidak ada sistem pelaporan yang memadai, sehingga sulit untuk melacak status setiap test case atau mengatur hasil berdasarkan modul/package.

2. Action

Membangun framework pengujian API otomatis yang komprehensif menggunakan Java + TestNG + Rest Assured:

  • Modul Auth: login valid & tidak valid, username/password kosong.
  • Modul Kategori Olahraga: buat, perbarui, hapus, daftar kategori.
  • Modul Aktivitas Olahraga: buat aktivitas, tetapkan kategori, daftar aktivitas.
  • Implementasi BaseTest dengan ExtentReports:
    • Laporan HTML tunggal untuk semua pengujian.
    • Penetapan kategori otomatis berdasarkan package (auth, sportCategory, sportActivity).
    • Logging otomatis dari request, response, dan status lulus/gagal untuk setiap test case.
  • Meningkatkan keamanan dan keterbacaan:
    • Masking username/password dalam log.
    • Menyimpan token login ke JSON untuk digunakan kembali dalam pengujian lain.
  • Mengatur struktur proyek yang modular untuk pemeliharaan dan skalabilitas yang mudah, termasuk utilitas (ConfigReader, ExtentManager) dan request body (LoginBody, SportCategoryBody).

3. Result

  • Laporan HTML tunggal profesional yang menunjukkan semua test case, status, request/response, dan dapat difilter berdasarkan kategori modul.
  • Framework otomatisasi mudah diperluas untuk modul baru tanpa menulis ulang kode pelaporan.
  • Menghemat lebih dari 70% waktu QA dibandingkan dengan pengujian manual.
  • Framework berfungsi sebagai proyek portofolio yang kuat, menunjukkan keterampilan dalam:
    • Pengujian otomatisasi API
    • Pelaporan pengujian profesional
    • Desain framework modular dan skalabel
    • Praktek terbaik seperti masking data sensitif, penggunaan kembali token, dan kategorisasi otomatis.

Tokopedia Exploratory & Accessibility Testing

Pengujian eksploratif dan aksesibilitas secara mendalam pada platform e-commerce Tokopedia. Proyek ini meliputi pengujian eksploratif untuk mengidentifikasi bug serta kendala tak terduga, serta pengujian aksesibilitas guna memastikan aplikasi dapat digunakan oleh semua kalangan, termasuk pengguna dengan disabilitas. Menerapkan metodologi pengujian yang sistematis dengan dokumentasi lengkap.

1. Problem

Situs web Tokopedia belum diuji untuk aksesibilitas dan responsivitas UI/UX. Sebagai salah satu platform e-commerce terbesar, Tokopedia harus memastikan semua komponen dapat digunakan oleh masyarakat umum dan orang dengan disabilitas.

2. Action

  • Melakukan pengujian eksploratif pada fitur: homepage, pencarian, detail produk, keranjang, checkout.
  • Menggunakan Axe DevTools & Chrome DevTools untuk mendeteksi pelanggaran WCAG 2.1 AA.
  • Menganalisis 13 bug (8 UI/UX, 5 aksesibilitas).
  • Menyiapkan laporan penutupan pengujian dengan rekomendasi untuk aria-label, kontras warna, dan validasi input.

3. Result

Selama Pengujian Eksploratif dan Aksesibilitas pada Tokopedia, saya menemukan total 13 bug yang mempengaruhi fungsionalitas, UI/UX, dan aksesibilitas. Pengujian menggabungkan metode manual dan alat seperti Axe DevTools, Chrome DevTools, dan panduan WCAG 2.1 AA.

Exploratory Testing

Temuan: 8 bug

Tingkat Keparahan: 3 Sedang, 5 Minor

Area terdampak: Homepage, Pencarian, Keranjang, Checkout, dan Halaman Produk

Temuan utama:

  • Gambar produk dan banner promosi sering gagal dimuat atau tumpang tindih.
  • Validasi input kurang optimal, misal: field nomor telepon memungkinkan nilai di luar batas wajar.
  • Pesan error tidak konsisten dengan kondisi input yang sebenarnya.
  • Ilustrasi dan visual tertentu tidak dimuat dengan benar di halaman akun.

Rekomendasi: Tingkatkan validasi input, optimalkan lazy loading gambar, dan jalankan pengujian regresi setelah pembaruan UI untuk menjaga stabilitas visual.

Accessibility Testing

Temuan: 5 bug

Tingkat Keparahan: 2 Tinggi, 2 Sedang, 1 Minor

Area terdampak: Keranjang, Checkout, Filter, dan Halaman Transaksi

Temuan utama:

  • Tombol dan ikon kurang teks deskriptif (aria-label), membuatnya tidak dapat diakses oleh screen reader.
  • Beberapa checkbox filter tidak memiliki label yang dapat dibaca oleh alat bantu.
  • Warna teks dan tombol memiliki kontras rendah (rasio di bawah 4.5:1).
  • Beberapa elemen interaktif kurang nama yang dapat diakses.

Rekomendasi: Tambahkan aria-label, role, dan teks deskriptif ke elemen interaktif. Tingkatkan kontras teks-latar belakang untuk memenuhi WCAG 2.1 AA, dan uji ulang dengan Axe DevTools dan Lighthouse untuk memastikan kepatuhan aksesibilitas.

API Testing with Postman and Database Validation

Inisiatif pengujian API secara mendalam menggunakan Postman untuk memvalidasi endpoint REST API serta validasi database guna menjamin integritas data secara menyeluruh. Mencakup simulasi berbagai metode HTTP (GET, POST, PUT, DELETE), validasi status code, skema JSON, hingga operasi database dengan menerapkan best practices pengujian API modern.

1. Problem

Selama integrasi API untuk proyek Sport Activity, ditemukan beberapa masalah yang mempengaruhi kualitas dan keandalan sistem backend.

Beberapa API endpoint menunjukkan:

  • Ketidakkonsistenan antara dokumentasi dan respons aktual.
  • Validasi input yang buruk (empty required fields still accepted).
  • Status kode yang salah (misal: 200 OK dikembalikan untuk permintaan yang gagal).
  • Kurangnya autentikasi pada endpoint sensitif seperti update dan delete.

Masalah-masalah ini dapat menyebabkan:

  • Data aktivitas olahraga yang tidak akurat.
  • Celah keamanan dan potensi penyalahgunaan API.
  • Penurunan kepercayaan pengguna terhadap sistem.

2. Action

Untuk mengatasi masalah ini, saya melakukan pengujian API manual yang menyeluruh di Postman dengan pengujian positif dan negatif.

Langkah-langkah pengujian meliputi:

  • Menganalisis dokumentasi API untuk memastikan parameter, metode, dan respons sesuai.
  • Membuat 27 manual test case yang mencakup endpoint:
    • Buat, Baca, Perbarui, Hapus Aktivitas Olahraga
    • Autentikasi (Login/Register)
    • Data Kota dan Kategori Olahraga
  • Memvalidasi status code dan response body menggunakan JSON Path Finder.
  • Menguji autentikasi dan otorisasi dengan token JWT Bearer.
  • Identifikasi bug dan mencatat hasil pengujian (8 test case gagal dari total 27).

Semua pengujian diatur ke dalam Postman Collection modular per endpoint, dengan skrip pra-permintaan dan variabel lingkungan untuk pengujian yang efisien.

3. Result

Hasil pengujian menunjukkan:

  • 27 test case dieksekusi, dengan 19 lulus (70%) dan 8 gagal (30%).

Bug utama yang ditemukan meliputi:

  • Endpoint Buat Aktivitas Olahraga tidak menolak field wajib yang kosong.
  • Endpoint Perbarui Aktivitas dapat diakses tanpa token autentikasi.
  • Hapus Aktivitas mengembalikan 200 OK meskipun ID tidak valid.
  • Dokumentasi API tidak sejalan dengan implementasi aktual (terutama untuk field city_id dan sport_category_id).
