Saucedemo Automation

Proyek automation testing untuk aplikasi web saucedemo.com menggunakan Selenium WebDriver dengan Java. Mengimplementasikan Page Object Model (POM) untuk framework otomatisasi pengujian yang mudah dipelihara dan dapat diskalakan.

1. Problem

Proses manual regression testing pada website e-commerce SauceDemo membutuhkan waktu yang lama dan rawan inkonsistensi hasil, terutama setelah adanya perubahan fitur seperti login, cart, dan checkout. Diperlukan solusi automation testing end-to-end agar validasi dapat dilakukan cepat, berulang, dan akurat.

2. Action

Saya membangun automation framework berbasis Java, Selenium, TestNG, dan Maven menggunakan pendekatan Page Object Model (POM) agar struktur kode modular dan mudah dikelola.

Langkah-langkah utama yang saya lakukan:

  • Mendesain arsitektur framework dengan struktur folder terpisah untuk pages, tests, dan utilities.
  • Menulis 19 automated test case yang mencakup seluruh fitur utama:
    • Login positif & negatif
    • Tambah dan hapus produk dari keranjang
    • Checkout single & multiple product
    • Validasi error message dan field kosong
    • Logout scenario
  • Menggunakan TestNG untuk pengaturan test suite, prioritas eksekusi, dan assert validation.
  • Mengimplementasikan screenshot otomatis untuk setiap hasil eksekusi.
  • Menghasilkan HTML report otomatis (allure report) yang menampilkan hasil Pass/Fail, durasi, dan screenshot.
  • Menggunakan WebDriverManager agar setup ChromeDriver otomatis tanpa konfigurasi manual.

3. Result

  • Semua 19 test case berhasil dijalankan dan 100% Passed.
  • Seluruh skenario fungsional website (login, cart, checkout, logout) tervalidasi tanpa error.
  • Eksekusi test otomatis selesai dalam waktu ±4 menit, dibanding manual test ±15 menit.
  • Framework terbukti stabil, reusable, dan mudah diperluas untuk regression test berikutnya.

Tokopedia Exploratory & Accessibility Testing

Pengujian eksplorasi dan aksesibilitas komprehensif pada platform e-commerce Tokopedia. Proyek ini mencakup pengujian eksplorasi untuk menemukan bug dan masalah tak terduga, serta pengujian aksesibilitas untuk memastikan aplikasi dapat diakses oleh pengguna disabilitas. Menerapkan pendekatan pengujian yang sistematis dengan dokumentasi yang komprehensif.

1. Problem

Website Tokopedia belum pernah diuji dari sisi aksesibilitas dan UI/UX responsiveness. Sebagai salah satu e-commerce terbesar, perlu dipastikan seluruh komponen bisa digunakan oleh pengguna umum maupun penyandang disabilitas.

2. Action

  • Melakukan exploratory test pada fitur: homepage, search, product detail, cart, checkout.
  • Menggunakan Axe DevTools & Chrome Inspect Element untuk mendeteksi pelanggaran WCAG 2.1 AA.
  • Menganalisis 13 bug (8 UI/UX, 5 accessibility).
  • Membuat test closure report dengan rekomendasi perbaikan aria-label, kontras warna, dan validasi input.

3. Result

Selama pelaksanaan Exploratory dan Accessibility Testing pada website Tokopedia, saya menemukan total 13 bug yang memengaruhi aspek fungsionalitas, tampilan UI/UX, dan aksesibilitas. Pengujian dilakukan menggunakan kombinasi metode manual dan alat bantu seperti Axe DevTools, Chrome Inspect Element, serta acuan WCAG 2.1 AA.

Exploratory Testing

Jumlah temuan: 8 bug

Tingkat keparahan: 3 Medium, 5 Minor

Area terdampak: Homepage, Search, Cart, Checkout, dan Product Page

Temuan utama:

  • Gambar produk dan banner promosi sering tidak tampil atau saling menimpa.
  • Validasi input belum optimal, seperti field nomor handphone yang bisa diisi melebihi batas wajar.
  • Pesan error tidak konsisten dengan kondisi input aktual.
  • Ilustrasi dan elemen visual tertentu tidak dimuat dengan benar pada halaman akun.

Rekomendasi: Perbaiki validasi input field, optimalkan lazy loading gambar, serta lakukan regression testing setelah update UI agar stabilitas visual tetap terjaga.

Accessibility Testing

Jumlah temuan: 5 bug

Tingkat keparahan: 2 High, 2 Medium, 1 Minor

Area terdampak: Cart, Checkout, Filter, dan Transaction Page

Temuan utama:

  • Tombol dan ikon tidak memiliki teks deskriptif (aria-label), sehingga tidak dapat diakses screen reader.
  • Beberapa checkbox filter tidak memiliki label yang bisa dibaca alat bantu.
  • Warna teks dan tombol memiliki kontras rendah (rasio di bawah 4.5:1).
  • Beberapa elemen interaktif tidak memiliki accessible name.

Rekomendasi: Tambahkan aria-label, role, dan teks deskriptif pada elemen interaktif. Tingkatkan rasio kontras warna teks–latar belakang sesuai standar WCAG 2.1 AA, serta uji ulang dengan Axe DevTools dan Lighthouse untuk memastikan kepatuhan aksesibilitas.

API Testing with Postman and Database Validation

Proyek pengujian API komprehensif menggunakan Postman untuk menguji endpoint REST API dan validasi database untuk memastikan integritas data. Proyek ini mencakup pengujian berbagai metode HTTP (GET, POST, PUT, DELETE), validasi kode respons, validasi skema, dan pengujian operasi database. Menerapkan praktik terbaik untuk pengujian API termasuk variabel lingkungan, kumpulan pengujian, dan skrip pengujian otomatis.

1. Problem

Selama tahap integrasi API pada proyek Sport Activity, ditemukan beberapa kendala yang mempengaruhi kualitas dan keandalan sistem backend.

Beberapa endpoint API menunjukkan:

  • Inkonsistensi antara dokumentasi dan response aktual.
  • Validasi input yang tidak optimal (field kosong masih diterima).
  • Kesalahan status code (misalnya 200 OK dikembalikan untuk request gagal).
  • Kurangnya mekanisme autentikasi pada endpoint sensitif seperti update dan delete.

Masalah ini berpotensi menyebabkan:

  • Ketidakakuratan data aktivitas olahraga.
  • Celah keamanan dan potensi penyalahgunaan API.
  • Penurunan kepercayaan pengguna terhadap sistem.

2. Action

Untuk mengatasi permasalahan tersebut, dilakukan proses manual API testing secara menyeluruh menggunakan Postman dengan pendekatan positive & negative testing.

Langkah-langkah pengujian mencakup:

  • Analisis dokumentasi API untuk memastikan kesesuaian parameter, metode, dan respons.
  • Pembuatan test case manual (27 skenario) mencakup endpoint:
    • Create, Read, Update, Delete Sport Activity
    • Authentication (Login/Register)
    • City dan Sport Category Data
  • Validasi status code dan response body menggunakan JSON Path Finder.
  • Uji autentikasi dan otorisasi dengan token Bearer JWT.
  • Identifikasi bug dan pelaporan hasil uji (8 test case gagal dari 27 total).

Semua pengujian dilakukan dalam Postman Collection yang disusun modular per endpoint, disertai pre-request script dan environment variable untuk efisiensi pengujian.

3. Result

Hasil pengujian menunjukkan:

  • 27 test case dijalankan, dengan 19 passed (70%) dan 8 failed (30%).

Bug utama yang ditemukan meliputi:

  • Endpoint Create Sport Activity tidak menolak field kosong meskipun wajib diisi.
  • Endpoint Update Activity masih dapat diakses tanpa token autentikasi.
  • Delete Activity mengembalikan status 200 OK meskipun ID tidak valid.
  • Dokumentasi API tidak sinkron dengan implementasi aktual (terutama pada field city_id dan sport_category_id).
