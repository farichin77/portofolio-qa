Tokopedia Exploratory & Accessibility Testing

Comprehensive exploratory and accessibility testing on the Tokopedia e-commerce platform. This project included exploratory testing to find bugs and unexpected issues, and accessibility testing to ensure the application is accessible for users with disabilities. Applied a systematic testing approach with comprehensive documentation.

1. Problem

The Tokopedia website had not been tested for accessibility and UI/UX responsiveness. As one of the largest e-commerce platforms, it must ensure all components are usable by the general public and people with disabilities.

2. Action

  • Performed exploratory testing on features: homepage, search, product detail, cart, checkout.
  • Used Axe DevTools & Chrome DevTools to detect WCAG 2.1 AA violations.
  • Analyzed 13 bugs (8 UI/UX, 5 accessibility).
  • Prepared a test closure report with recommendations for aria-labels, color contrast, and input validation.

3. Result

During the Exploratory and Accessibility Testing on Tokopedia, I found a total of 13 bugs affecting functionality, UI/UX, and accessibility. Testing combined manual methods and tools such as Axe DevTools, Chrome DevTools, and WCAG 2.1 AA guidelines.

Exploratory Testing

Findings: 8 bugs

Severity: 3 Medium, 5 Minor

Affected areas: Homepage, Search, Cart, Checkout, and Product Page

Key findings:

  • Product images and promotional banners often failed to render or overlapped.
  • Input validation was suboptimal, e.g., phone number field allowed values beyond reasonable limits.
  • Error messages were inconsistent with actual input conditions.
  • Certain illustrations and visuals did not load correctly on the account page.

Recommendation: Improve input validation, optimize image lazy loading, and run regression testing after UI updates to maintain visual stability.

Accessibility Testing

Findings: 5 bugs

Severity: 2 High, 2 Medium, 1 Minor

Affected areas: Cart, Checkout, Filter, and Transaction Page

Key findings:

  • Buttons and icons lacked descriptive text (aria-label), making them inaccessible to screen readers.
  • Some filter checkboxes had no labels readable by assistive tools.
  • Text and button colors had low contrast (ratio below 4.5:1).
  • Some interactive elements lacked an accessible name.

Recommendation: Add aria-labels, roles, and descriptive text to interactive elements. Increase text–background contrast to meet WCAG 2.1 AA, and retest with Axe DevTools and Lighthouse to ensure accessibility compliance.

API Testing with Postman and Database Validation

Comprehensive API testing project using Postman to test REST API endpoints and database validation to ensure data integrity. Covered HTTP methods (GET, POST, PUT, DELETE), status code validation, schema validation, and database operations, applying API testing best practices such as environment variables, collections, and automated test scripts.

1. Problem

During API integration for the Sport Activity project, several issues were found that affected the quality and reliability of the backend system.

Some API endpoints showed:

  • Inconsistency between documentation and actual responses.
  • Poor input validation (empty required fields still accepted).
  • Incorrect status codes (e.g., 200 OK returned for failed requests).
  • Lack of authentication on sensitive endpoints such as update and delete.

These issues could lead to:

  • Inaccurate sport activity data.
  • Security gaps and potential API abuse.
  • Decreased user trust in the system.

2. Action

To address these issues, I conducted thorough manual API testing in Postman with both positive and negative testing.

Testing steps included:

  • Analyzed API documentation to ensure parameters, methods, and responses matched.
  • Created 27 manual test cases covering endpoints:
    • Create, Read, Update, Delete Sport Activity
    • Authentication (Login/Register)
    • City and Sport Category data
  • Validated status codes and response bodies using JSON Path Finder.
  • Tested authentication and authorization with JWT Bearer tokens.
  • Identified bugs and recorded test results (8 failed test cases out of 27 total).

All tests were organized into a modular Postman Collection per endpoint, with pre-request scripts and environment variables for efficient testing.

3. Result

Test results showed:

  • 27 test cases executed, with 19 passed (70%) and 8 failed (30%).

Key bugs found included:

  • Create Sport Activity endpoint did not reject empty required fields.
  • Update Activity endpoint was accessible without an authentication token.
  • Delete Activity returned 200 OK even when the ID was invalid.
  • API documentation was not aligned with actual implementation (notably for fields city_id and sport_category_id).
